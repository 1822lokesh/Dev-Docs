# Introduction to React.js

## What is React?
React is an open-source JavaScript library used for building user interfaces (UI), specifically for single-page applications. It was created by Meta (Facebook) and is maintained by a large community of developers.

Unlike traditional web development where you build entire pages, React allows you to build **Components**—small, isolated pieces of code that represent parts of the interface (like a button, a search bar, or a navigation menu).

## Why we Use React?
React is currently the most popular front-end library in the world. Here is why:

* **Component-Based:** You write code once and reuse it everywhere.
* **Declarative:** You describe what the UI should look like based on the current data, and React handles the "how" (updating the DOM).
* **Virtual DOM:** React is fast because it uses a "Virtual DOM" to minimize expensive operations in the actual browser.
* **Huge Ecosystem:** Since it's so popular, there are endless libraries for routing, styling, and state management.

## How does ReactJS Work?

Before understanding how React works, let us see the comparision between Real DOM and Virtual DOM.

### Virtual DOM vs Real DOM 

### What is **Real DOM**

**Real DOM** is the **actual Document Object Model created by the browser** to represent the structure of a web page.

It is a **tree-like structure of HTML elements** which the browser uses to render UI on the screen.

Whenever we change something in UI using normal JavaScript,
the **browser updates the Real DOM directly**.


### Example

```html
<div>
  <h1>Hello</h1>
</div>
```

Browser converts this into a DOM tree internally.

If we update text using JavaScript:

```js
document.querySelector("h1").innerText = "Hi";
```

Browser will:

* Find element
* Update DOM
* Recalculate layout
* Repaint UI

Interview Point:

> Real DOM manipulation is **slow and performance expensive** because every change triggers reflow and repaint.

---

## What is **Virtual DOM**

**Virtual DOM** is a **lightweight JavaScript object representation of the Real DOM** maintained in memory by React.

React uses Virtual DOM to **optimize UI updates and improve performance**.

Instead of updating Real DOM directly, React:

1. Updates Virtual DOM
2. Compares with previous Virtual DOM
3. Updates only changed parts in Real DOM

This process is called **Reconciliation**.

---

### Example Concept

Initial UI:

```jsx
<h1>0</h1>
```

After state update:

```jsx
<h1>1</h1>
```

React compares both Virtual DOM trees
Finds only text change
Updates only that node in Real DOM

Interview Point:

> Virtual DOM makes React applications **fast and efficient** by minimizing direct DOM operations.

---

## Key Differences (Important Interview Table)

| Feature       | Real DOM              | Virtual DOM               |
| ------------- | --------------------- | ------------------------- |
| Definition    | Actual browser DOM    | Virtual copy in memory    |
| Update Speed  | Slow                  | Fast                      |
| Update Method | Direct manipulation   | Diffing & reconciliation  |
| Performance   | Less efficient        | Highly optimized          |
| UI Re-render  | Whole tree may update | Only changed nodes update |
| Used By       | Vanilla JS, jQuery    | React                     |

---

## Interview Answer

**Real DOM is the actual browser structure of a webpage, while Virtual DOM is a lightweight in-memory copy used by React to efficiently update only the changed UI elements.**

## Working of React


**React.js** is a **JavaScript library for building user interfaces (UI)**, especially **single-page applications (SPA)** where content updates dynamically without reloading the page.

It follows a **component-based architecture**, meaning the UI is divided into **small reusable pieces called components**.

---

### Core Working Principle (High-Level Flow)

React works mainly using **Virtual DOM + Component Rendering + State Updates + Efficient Reconciliation**.

The complete flow:

```
User Action → State Change → Virtual DOM Update → Diffing → Real DOM Update
```

---

### Step-by-Step Explanation

### 1️⃣ Component Based UI

In React, the UI is divided into **independent reusable components**.

Example:

```jsx
function App() {
  return (
    <div>
      <Header />
      <Sidebar />
      <Content />
    </div>
  );
}
```

Interview Point:
React follows **modular architecture**, improving code maintainability and reusability.

---

### 2️⃣ Virtual DOM Concept

React does NOT directly update the **Real DOM** (browser DOM).
Instead, it creates a **Virtual DOM (lightweight copy of real DOM in memory)**.

Why?

* Real DOM updates are **slow**
* Virtual DOM updates are **fast**

Interview Definition:

> Virtual DOM is a lightweight JavaScript representation of the real DOM used by React to optimize UI updates.

---

### 3️⃣ State Change Triggers Re-Rendering

Whenever **state or props change**, React:

1. Creates a **new Virtual DOM tree**
2. Compares it with the **previous Virtual DOM**

Example:

```jsx
function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <div>
      <h1>{count}</h1>
      <button onClick={() => setCount(count + 1)}>Increase</button>
    </div>
  );
}
```

When button clicked → state updates → component re-renders.

Interview Point:
React UI is **state-driven**.

---

### 4️⃣ Diffing Algorithm (Reconciliation Process)

React compares:

Old Virtual DOM
New Virtual DOM

This process is called **Reconciliation**.

React finds **only the changed elements**.

Example:

```
Old → <h1>0</h1>
New → <h1>1</h1>
```

React updates **only text node**, not whole DOM.

Interview Point:
React uses **efficient diffing algorithm (O(n))** to improve performance.

---

### 5️⃣ Efficient Real DOM Update

After finding differences, React updates **only required parts** in Real DOM.

This is called **Selective Rendering / Efficient DOM Manipulation**

Result:

* Faster UI updates
* Better performance
* Smooth user experience

---

## One Line Interview Answer

**React works by creating a Virtual DOM, comparing it with previous UI using reconciliation, and updating only the changed parts in the Real DOM for efficient rendering.**

---

## Real-World Analogy

Think like **Google Docs live editing**

Instead of refreshing whole page
Only changed text updates instantly

# Reconciliation in React

**Reconciliation is the process through which React updates the UI efficiently by comparing the previous Virtual DOM with the new Virtual DOM and updating only the changed elements in the Real DOM.**

It is also called **Diffing Algorithm Process**.

---

## Why Reconciliation is Needed

Direct DOM updates are **slow and costly**.

So React:

* Creates new Virtual DOM after state/props change
* Compares with old Virtual DOM
* Finds minimal changes
* Updates only required nodes

Result → **High Performance UI Rendering**

---

## Working Steps of Reconciliation

### 1️⃣ State or Props Change

Whenever:

* `setState()`
* `useState setter`
* Parent re-render

React triggers re-render.

---

### 2️⃣ New Virtual DOM Creation

React creates **new Virtual DOM tree**.

Example:

```jsx
<h1>0</h1>
```

After update:

```jsx
<h1>1</h1>
```

---

### 3️⃣ Diffing Algorithm

React compares:

Old Virtual DOM
New Virtual DOM

Rules React follows:

* Different element type → destroy & recreate
* Same element type → update only attributes
* List updates handled using **keys**

Interview Point:

> React diffing algorithm works in **O(n) time complexity**.

---

### 4️⃣ Real DOM Update

After finding difference →
React updates **only changed node**

This is called **Selective DOM Rendering**

---

## Interview ans
**Reconciliation is the process in which React compares the old and new Virtual DOM using a diffing algorithm and updates only the necessary parts of the Real DOM to improve performance.**

# Component Lifecycle / Rendering Flow

Lifecycle means **different stages of a component from creation to removal**.

In **modern React (Functional Components)** lifecycle is handled using **Hooks**.

---

## Main Lifecycle Phases

### 1. Mounting Phase

Component is **created and inserted into DOM**

Steps:

* Function component executes
* JSX returned
* UI rendered on screen

Example:

```jsx
useEffect(() => {
  console.log("Component Mounted");
}, []);
```

Interview Point:

> Empty dependency array → runs only once after first render.

---

### 2. Updating Phase

Happens when:

* State changes
* Props changes
* Parent re-renders

Example:

```jsx
useEffect(() => {
  console.log("Component Updated");
}, [count]);
```

Runs whenever `count` changes.

---

### 3. Unmounting Phase

Component removed from DOM.

Example:

```jsx
useEffect(() => {
  return () => {
    console.log("Component Unmounted");
  };
}, []);
```

Used for:

* Clearing timers
* Removing event listeners
* API cleanup

---

## Complete Rendering Flow 

```text
User Action
   ↓
State Update
   ↓
Component Re-render
   ↓
New Virtual DOM
   ↓
Reconciliation (Diffing)
   ↓
Real DOM Update
   ↓
UI Updated on Screen
```

---

### Interview Answer

**Component lifecycle in React represents different stages such as mounting, updating, and unmounting, which control how a component renders and behaves during its existence in the UI.**

# Framework vs. Library

In software development, we rarely write everything from scratch. We use external code to help us. Two common types of external code are **Libraries** and **Frameworks**.

## What is a Library?
A library is a collection of pre-written code (functions, classes, or components) that you can use to perform specific tasks.

* **Key Characteristic:** You are in control. You decide when to call the library and where to use it in your code.
* **Analogy:** A Library is like going to a **supermarket**. You pick the ingredients you want (vegetables, spices, meat) and you decide how to cook them and what dish to make.
* **Example:** React, jQuery, Lodash.

## What is a Framework?
A framework is a comprehensive platform that provides a foundation for building software. It includes libraries, but it also dictates how your project should be structured.

* **Key Characteristic:** The Framework is in control. It provides a skeleton, and you fill in the blanks. The framework calls your code when it needs to.
* **Analogy:** A Framework is like ordering a **HelloFresh meal kit**. The menu is decided, the ingredients are pre-measured, and you must follow the specific recipe card to get the result.
* **Example:** Angular, Vue (often considered a progressive framework), Next.js, Django.

## The Key Difference: "Inversion of Control" (IoC)
The technical difference between the two is called **Inversion of Control**.

* **With a Library:** Your application code calls the library.
    * `Your Code -> calls -> Library`
* **With a Framework:** The framework calls your application code.
    * `Framework -> calls -> Your Code`

[Image of Inversion of Control Diagram Framework vs Library]

## Comparison Table

| Feature | Library | Framework |
| :--- | :--- | :--- |
| **Control** | You control the flow (You call the code). | The Framework controls the flow (It calls your code). |
| **Flexibility** | High. You can swap libraries easily. | Low. You are stuck with the framework's rules. |
| **Opinionated?** | No. It doesn't care how you structure files. | Yes. It enforces specific file structures and patterns. |
| **Replacement** | Easy to replace or upgrade specific parts. | Hard. Rewriting the app is often required to switch frameworks. |
| **Example** | React.js (UI only) | Angular (UI + Routing + HTTP + State) |

## Why is React considered a Library?
React is a library because it only cares about one thing: **The User Interface** (rendering views).

* React doesn't tell you how to handle routing (navigating between pages).
* React doesn't tell you how to make HTTP requests (fetching data).
* React doesn't force a specific folder structure.

Because of this, you have to choose other libraries to work with React (like `react-router-dom` for pages or `axios` for API calls).
