# SQL Joins

### SQL Joins are used to combine data from two or more tables based on a related column. 

### They help in:

* Retrieving connected data stored across multiple tables.
* Matching records using common columns.
* Improving data analysis by combining related information.
* Creating meaningful result sets from separate tables.

## 1. INNER JOIN

### INNER JOIN is used to retrieve rows where matching values exist in both tables.

### It helps in:

* Combining records based on a related column.
* Returning only matching rows from both tables.
* Excluding non-matching data from the result set.
* Ensuring accurate data relationships between tables.


**Note: We can also write JOIN instead of INNER JOIN.  JOIN is same as INNER JOIN.** 

## INNER JOIN Example

### Problem

Fetch the employee name and the department name they belong to.

---

### Query

```sql
SELECT emp_name , dept_name
FROM Employee e
INNER JOIN Department d 
ON e.dept_id = d.dept_id;
```

---

### How It Works

* `Employee e` → alias **e** is used for Employee table

* `Department d` → alias **d** is used for Department table

* `INNER JOIN`:

  * Returns only **matching records** from both tables
  * If there is no match → row is **excluded**

* `ON e.dept_id = d.dept_id`:

  * This is the **join condition**
  * It connects both tables using `dept_id`

---

### Output 

| emp_name | dept_name  |
| -------- | ---------- |
| Lokesh   | HR         |
| Ravi     | IT         |
| Anu      | Finance    |
| Kiran    | IT         |
| Meena    | Marketing  |
| Seena    | HR         |
| Arjun    | Operations |

---

### Key Points

* Only employees with a **valid department match** are shown
* Department `EEE (70)` is **not included** → no employees mapped
* Every employee in your data has a matching department → so all 7 rows appear
* If any employee had an invalid `dept_id`, they would be **excluded**

---

### Simple Understanding

INNER JOIN = **Intersection of two tables (common data only)**

---


## 2. LEFT JOIN

### LEFT JOIN (also called LEFT OUTER JOIN) retrieves all records from the left table and only the matching records from the right table. If no match is found in the right table, the query will return NULL values for its columns.

* Returns all rows from the left table.
* Includes only matching rows from the right table.
* Non-matching rows in the right table are represented as NULL.

## LEFT JOIN Example

### Problem

Fetch all employee names and the department names they belong to.

---

### Query

```sql id="p9x3dl"
SELECT emp_name , dept_name
FROM Employee e
LEFT JOIN Department d 
ON e.dept_id = d.dept_id;
```

---

### How It Works

* `LEFT JOIN`:

  * Returns **all records from the left table (Employee)**
  * Returns matching records from the right table (Department)
  * If no match → fills with **NULL**

* `Employee` = **Left Table**

* `Department` = **Right Table**

* `ON e.dept_id = d.dept_id`:

  * Matches employees with their departments

---

### Output 

| emp_name | dept_name  |
| -------- | ---------- |
| Lokesh   | HR         |
| Ravi     | IT         |
| Anu      | Finance    |
| Kiran    | IT         |
| Meena    | Marketing  |
| Seena    | HR         |
| Arjun    | Operations |

---

### Key Points

* All employees are **always included** (even if no department exists)
* If an employee had an invalid `dept_id` → `dept_name` would be **NULL**
* Department `EEE (70)` is **not shown** → because LEFT JOIN prioritizes Employee table
* Output looks same as INNER JOIN here because all employees have valid departments

---

### Simple Understanding

LEFT JOIN = **LEFT JOIN = INNER JOIN + any additional records from the left table**

If no match → **NULL on right side**

---

 ## RIGHT JOIN

 ###  RIGHT JOIN (also called RIGHT OUTER JOIN) is used to combine rows from two tables based on a related column. It returns all records from the right table and only the matching records from the left table. If there is no match in the left table, the result will show NULL values for the left table’s columns.

* Returns all rows from the right table.
* Includes only matching rows from the left table.
* Non-matching rows from the left table appear as NULL.
## RIGHT JOIN Example

### Problem

Fetch all employee names and the department names they belong to (including departments with no employees).

---

### Query

```sql id="k3n8xq"
SELECT emp_name , dept_name
FROM Employee e
RIGHT JOIN Department d 
ON e.dept_id = d.dept_id;
```

---

### How It Works

* `RIGHT JOIN`:

  * Returns **all records from the right table (Department)**
  * Returns matching records from the left table (Employee)
  * If no match → fills with **NULL**

* `Employee` = **Left Table**

* `Department` = **Right Table**

* `ON e.dept_id = d.dept_id`:

  * Matches employees with their departments

---

### Output 

| emp_name | dept_name  |
| -------- | ---------- |
| Lokesh   | HR         |
| Seena    | HR         |
| Ravi     | IT         |
| Kiran    | IT         |
| Anu      | Finance    |
| Meena    | Marketing  |
| Arjun    | Operations |
| NULL     | EEE        |

---

###  Key Points

* All departments are **always included**
* Department `EEE (70)` appears even though **no employees exist**
* For unmatched department → `emp_name` becomes **NULL**
* Multiple employees in same department → multiple rows (e.g., HR, IT)
* Output may have **more rows than Employee table**

---

###  Simple Understanding

RIGHT JOIN = **INNER JOIN + any additional records from the right table**

If no match → **NULL on left side**

---

## Multi-Table JOIN Example (Employee + Department + Manager + Project)

### Problem

Fetch details of **all employees**, their **manager**, their **department**, and the **projects they work on**.

---

### Query

```sql id="q7m2ka"
SELECT 
    e.emp_name,
    d.dept_name,
    m.manager_name,
    p.project_name
FROM Employee e
LEFT JOIN Department d ON e.dept_id = d.dept_id
LEFT JOIN Manager m ON e.manager_id = m.manager_id
LEFT JOIN Project p ON e.emp_id = p.team_mem_id;
```

---

###  How It Works

* `Employee` → **Base (driving) table**

---

### 🔹 Step 1: LEFT JOIN Department

* Keeps **all employees**
* Adds department if match exists
* If not → `dept_name = NULL`

---

### 🔹 Step 2: INNER JOIN Manager

* Keeps only employees with **valid manager_id**
* If no matching manager → employee is **removed**

---

### 🔹 Step 3: LEFT JOIN Project

* Keeps all remaining employees
* Adds project details if available
* If no project → `project_name = NULL`

---

###  Output (Based on Your Data)

| emp_name | dept_name  | manager_name | project_name  |
| -------- | ---------- | ------------ | ------------- |
| Lokesh   | HR         | Alice        | AI System     |
| Ravi     | IT         | Bob          | Web App       |
| Ravi     | IT         | Bob          | Data Pipeline |
| Anu      | Finance    | Charlie      | NULL          |
| Kiran    | IT         | Bob          | NULL          |
| Meena    | Marketing  | Charlie      | NULL          |
| Seena    | HR         | Alice        | Security Tool |
| Arjun    | Operations | Alice        | NULL          |

---

###  Key Points

* **All employees are included** (because starting table is Employee)
* `INNER JOIN Manager` ensures:

  * Every employee must have a **valid manager**
* `LEFT JOIN Project`:

  * Employees without projects still appear with **NULL**
* **Duplicate rows occur**:

  * Ravi appears twice → works on 2 projects
* Projects table controls **row multiplication**
* Join order matters:

  * INNER JOIN can **filter out rows before next joins**

 ## FULL JOIN

 ### The FULL JOIN (or FULL OUTER JOIN) in SQL returns all rows from both tables, combining matched rows and filling unmatched rows with NULL values. It is basically the combination of LEFT JOIN and RIGHT JOIN

* Retrieves all rows from both tables.
* Matches rows where conditions meet.
* Fills NULLs where no match exists.
* Combines results of LEFT JOIN and RIGHT JOIN.

## FULL JOIN Example

###  Problem

Fetch employee names and department names, including:

* Employees without departments
* Departments without employees

---

###  Query

```sql id="z8n4qp"
SELECT emp_name, dept_name
FROM Employee e
FULL JOIN Department d 
ON d.dept_id = e.dept_id;
```

---

###  How It Works

* `FULL JOIN` (or FULL OUTER JOIN):

  * Returns **all records from both tables**
  * Matches where possible
  * If no match → fills with **NULL**

* Combines:

  * **LEFT JOIN behavior** (all employees)
  * **RIGHT JOIN behavior** (all departments)

---

### Output (Based on Your Data)

| emp_name | dept_name  |
| -------- | ---------- |
| Lokesh   | HR         |
| Seena    | HR         |
| Ravi     | IT         |
| Kiran    | IT         |
| Anu      | Finance    |
| Meena    | Marketing  |
| Arjun    | Operations |
| NULL     | EEE        |

---

### Key Points

* Includes **all employees** (even if no department)
* Includes **all departments** (even if no employees)
* Department `EEE (70)` appears with **NULL employee**
* If an employee had no matching department → `dept_name` would be **NULL**
* Can produce **NULLs on both sides**

---

###  Simple Understanding

 FULL JOIN = **All data from both tables**

If no match:

* Left side missing → NULL
* Right side missing → NULL

---

### Important Note

* Not all databases support `FULL JOIN` directly (e.g., MySQL)
* Alternative in MySQL:

```sql id="y3k1ts"
SELECT emp_name, dept_name
FROM Employee e
LEFT JOIN Department d ON e.dept_id = d.dept_id

UNION

SELECT emp_name, dept_name
FROM Employee e
RIGHT JOIN Department d ON e.dept_id = d.dept_id;
```

---

### One-line takeaway

FULL JOIN = **LEFT JOIN + RIGHT JOIN combined**

## CROSS JOIN
### * SQL CROSS JOIN creates a Cartesian product of two tables, meaning it returns every possible combination of rows from both tables. 
### * It does not use a join condition, and the total number of rows in the result is the number of rows in the first table multiplied by the number of rows in the second table. 
### * It is useful for generating all combinations of data.

##  CROSS JOIN Example (with Company Table)

###  Problem

Fetch:

* Employee name
* Department name
* Company name
* Company location

---

## Step 1: Create Company Table

```sql id="c1q9ws"
CREATE TABLE Company (
    company_id INT PRIMARY KEY,
    company_name VARCHAR(50),
    location VARCHAR(50)
);
```

---

## Step 2: Insert One Record

```sql id="n8x2pl"
INSERT INTO Company VALUES
(1, 'TechCorp', 'Bangalore');
```

---

##  Query

```sql id="k4m8zx"
SELECT emp_name, dept_name, company_name, location
FROM Employee e
INNER JOIN Department d 
    ON e.dept_id = d.dept_id
CROSS JOIN Company c;
```

---

##  How It Works

### 🔹 INNER JOIN (Employee + Department)

* Fetches employees with their departments
* Only matching records are included

---

### 🔹 CROSS JOIN (with Company)

* Combines **every row with every row** from Company
* Since Company has **only 1 row**:
   That row is added to **all employees**

---

##  Output

| emp_name | dept_name  | company_name | location  |
| -------- | ---------- | ------------ | --------- |
| Lokesh   | HR         | TechCorp     | Bangalore |
| Ravi     | IT         | TechCorp     | Bangalore |
| Anu      | Finance    | TechCorp     | Bangalore |
| Kiran    | IT         | TechCorp     | Bangalore |
| Meena    | Marketing  | TechCorp     | Bangalore |
| Seena    | HR         | TechCorp     | Bangalore |
| Arjun    | Operations | TechCorp     | Bangalore |

---

##  Key Points

* `CROSS JOIN` = **Cartesian Product**
* If Company had:

  * 1 row → same company info repeated
  * 3 rows → each employee appears **3 times**
* No join condition is required for CROSS JOIN
* Useful when adding **common/static data** (like company info)


---

## Simple Understanding

CROSS JOIN = **Attach every row of one table to every row of another**

Here:

* 7 employees × 1 company = **7 rows**

---

###  One-line takeaway

CROSS JOIN is useful when you want to **add constant/global data to all rows**

## Natural Join

### Natural Join is a type of join that automatically matches and combines rows from two tables based on columns with same name and compatible data types. 
### Unlike other joins, you don’t need to manually specify condition; SQL detects common columns and returns a result without redundant duplicate columns.


##  NATURAL JOIN Example

###  Problem

Fetch employee names and department names using **NATURAL JOIN**.

---

###  Query

```sql id="m4k9zp"
SELECT emp_name, dept_name
FROM Employee
NATURAL JOIN Department;
```

---

##  How It Works

* `NATURAL JOIN`:

  * Automatically joins tables based on **columns with the same name**
  * No need to write `ON` condition

---

###  In Your Tables

Both tables have a common column:

* `dept_id` (in Employee)
* `dept_id` (in Department)

So SQL internally does:

```sql id="t8x2bn"
ON Employee.dept_id = Department.dept_id
```

---

##  Output (Based on Your Data)

| emp_name | dept_name  |
| -------- | ---------- |
| Lokesh   | HR         |
| Ravi     | IT         |
| Anu      | Finance    |
| Kiran    | IT         |
| Meena    | Marketing  |
| Seena    | HR         |
| Arjun    | Operations |

---

##  Key Points

* Works like an **INNER JOIN**
* Only returns **matching records**
* Automatically detects **common column names**
* You don’t need to specify join condition
* If no matching column exists →  error

---

##  Important Warning

* If multiple columns have same names → all are used for joining
* Can lead to **unexpected results**
* Less control compared to `INNER JOIN`

Example:
If both tables had `dept_id` and `manager_id` → both would be used

---

##  Simple Understanding

NATURAL JOIN = **Auto INNER JOIN using same column names**

---

###  One-line takeaway

 NATURAL JOIN saves typing, but **can be risky in real-world queries**

---

## SELF JOIN

### SQL Self Join is used when a table needs to be joined with itself to compare rows within the same table. It helps in finding relationships between records in a single table by treating it as two separate instances using table aliases.

## Applications of SQL Self Join
### SQL Self Join is widely used in different scenarios, such as:

* Hierarchical Data: Representing organizational structures like employee–manager, category–subcategory or parent–child relationships.

* Finding Relationships: Identifying relationships within the same table, for example linking friends in a social network or mapping task dependencies.

* Data Comparison: Comparing rows within same table, such as salaries of employees in same department.

* Detecting Duplicates: Finding duplicate records in a table by joining it with itself on duplicate criteria.

* Sequential Data Analysis: Comparing current rows with previous or next rows, useful for analyzing trends in sales or time-series data.

**Note: A self join is not a new join type. It simply means joining the same table twice with aliases.**

##  SELF JOIN Example (Family Table)

###  Problem

Fetch:

* Child name
* Child age
* Parent name
* Parent age

---

##  Query (SELF JOIN)

```sql id="n4c9jw"
SELECT 
    c.name AS child_name,
    c.age AS child_age,
    p.name AS parent_name,
    p.age AS parent_age
FROM Family c
JOIN Family p
ON c.parent_id = p.mem_id;
```

---

##  How It Works

* Same table is used **twice**

* Aliases:

  * `c` → child
  * `p` → parent

* `ON c.parent_id = p.mem_id`:

  * Links child to their parent

---

##  Output

| child_name | child_age | parent_name | parent_age |
| ---------- | --------- | ----------- | ---------- |
| Lokesh     | 30        | Ramesh      | 60         |
| Ravi       | 28        | Ramesh      | 60         |
| Anu        | 25        | Suresh      | 58         |
| Meena      | 22        | Suresh      | 58         |
| Kiran      | 5         | Lokesh      | 30         |
| Arjun      | 3         | Lokesh      | 30         |

---

##  Key Points

* SELF JOIN = joining a table with itself
* Required when data is **hierarchical** (parent-child)
* `parent_id = NULL` → top-level members (no parent)
* Those NULL parent rows are **not included** (due to INNER JOIN)
* Can create multiple levels (parent → child → grandchild)

---

##  Simple Understanding

 SELF JOIN = **Use same table as two roles (child & parent)**

---

###  One-line takeaway

 SELF JOIN is used when a table has **relationships within itself**
